# RTL Design AI Skill: 自仿真与验证

## 目的
与AI协作生成**Testbench和仿真激励**，进行功能验证，评估设计性能。

---

## 前置条件
- [ ] RTL代码已生成并通过代码审查
- [ ] 仿真环境已搭建（Modelsim / VCS / Vivado 等）
- [ ] 设计的关键测试场景已明确
- [ ] 性能评估指标已定义

---

## 📋 核心工作流

### 步骤 1: Testbench生成请求

**工程师向AI提供信息**：

```
你是一名资深的RTL验证工程师。现在基于已生成的RTL代码，我需要生成Testbench进行功能验证。

【设计信息】
- 模块名：[xxx_module]
- 功能简述：[简要功能描述]
- 已评审的架构文档：[参考]
- RTL代码位置：[代码路径]

【接口信息】
顶层接口：
  - 输入信号：[列表]
  - 输出信号：[列表]
  - 时钟/复位：clk, rst_n (active-low)
  - 频率：[xxx MHz]

【测试场景】
请生成针对以下场景的测试用例：

1. 基本功能测试
   场景1.1：正常读操作，数据=0x12345678
   场景1.2：正常写操作，数据=0xABCDEF00
   场景1.3：连续读写操作

2. 边界条件测试
   场景2.1：最大深度 FIFO 的满/空条件
   场景2.2：最大地址访问
   场景2.3：最小数据宽度

3. 异常处理测试
   场景3.1：写满时继续写的行为
   场景3.2：读空时继续读的行为
   场景3.3：同时复位和操作时的行为

4. 时序关键路径测试
   场景4.1：最坏case组合逻辑延迟验证
   场景4.2：多路同时请求的仲裁

5. 特殊场景
   场景5.1：[其他特殊场景]

【仿真环境】
- 语言：Verilog
- 仿真工具：Modelsim / Vivado (推荐用哪个？)
- Testbench风格：简单的 Verilog testbench / Cocotb
- 验证方法：[自定义关键点检查 / 或给出覆盖率要求]

【性能评估指标】
需要测量以下指标：
- 吞吐量：transactions/cycle
- 延迟：从请求到响应的 cycle 数
- 时序验证：最快能达到的频率

【请执行以下任务】
1. 生成完整的 Testbench 代码
2. 针对上述每个测试场景生成测试激励
3. 给出断言或检查点（关键信号的预期值）
4. 设计性能测量机制
5. 提供仿真运行方法和结果分析指南
```

### 步骤 2: AI生成Testbench

**AI输出内容**（预期）：
```
✓ 完整的 Testbench 顶层代码
✓ 测试激励生成模块（Task / Function）
✓ 关键信号检查和断言
✓ 性能计数器和测量逻辑
✓ 每个测试场景的详细注释
✓ 仿真运行脚本（do 文件或 shell 脚本）
✓ 预期的仿真结果示例
```

### 步骤 3: 工程师仿真验证 - 第一轮

**执行仿真**：

#### 3.1 编译和仿真
```bash
# 以 Modelsim 为例
vlog rtl/module_*.v tb/testbench_*.v
vsim -gui work.testbench_top -do "run -all"
```

#### 3.2 检查仿真结果
- [ ] 仿真是否正常完成（无编译错误、运行错误）？
- [ ] RTL 是否按预期执行？
- [ ] 关键信号的波形是否符合预期？
- [ ] 断言是否都通过？
- [ ] 有无意外的 warning？

#### 3.3 性能评估
- [ ] 吞吐量达到设计目标了吗？（目标：xxx trans/cycle）
- [ ] 延迟是否满足要求？（目标：xxx cycles）
- [ ] 关键路径是否如预期？

#### 3.4 覆盖率分析（如需要）
```bash
# 生成覆盖率报告
coverage report -detail
```
- [ ] 代码覆盖率是否达到目标？（目标：>85%）
- [ ] 功能覆盖率是否达到目标？（目标：>90%）
- [ ] 有哪些分支未被覆盖？是否需要补充测试？

**仿真问题反馈给AI**：
```
我运行了仿真，发现以下问题：

【功能错误】
1. 信号 xxx 在 time=xxx ns 时的值与预期不符
   预期：[xxxx]
   实际：[xxxx]
   原因分析：可能是 xxxx 逻辑有问题

2. 断言 "assert_xxx" 在 time=xxx ns 时失败
   失败条件：[xxxx]

【Testbench问题】
1. Testbench 中的激励生成有问题：
   问题描述：[xxxx]

【性能问题】
1. 吞吐量未达到设计目标：
   预期：10 trans/cycle
   实际：8 trans/cycle
   可能原因：[xxxx]

请修改 RTL 代码或 Testbench，解决上述问题。
```

### 步骤 4: RTL改进或Testbench修正

**若是RTL代码问题**：
- AI修改RTL代码，并解释改进
- 回到步骤3重新仿真

**若是Testbench问题**：
- AI修改Testbench激励或断言
- 回到步骤3重新仿真

重复直至：
- [ ] 所有测试用例通过
- [ ] 性能指标达到或接近目标
- [ ] 覆盖率满足要求

### 步骤 5: 压力测试和极限场景

一旦基本功能通过，进行进一步测试：

```
现在请进行压力测试和极限场景验证：

【压力测试】
1. 最大吞吐量测试：连续发送最大数量的请求，检查是否有数据丢失
2. 长时间运行：运行 1M+ cycles，检查是否有不稳定现象
3. 随机激励：生成随机的输入激励，检查是否有隐藏的 bug

【极限场景】
1. 极端参数配置：如最大FIFO深度、最大地址范围等
2. 特殊时序：如连续的同周期读写、复位中间的操作等
3. 边界转移：从满到空、从空到满的快速转移

请生成这些测试用例，并评估设计的鲁棒性。
```

### 步骤 6: 性能分析与优化建议

```
基于仿真结果，请分析以下内容：

1. 性能瓶颈分析
   - 关键路径在哪里？
   - 是什么限制了吞吐量？
   
2. 优化建议
   - 代码中是否有明显的优化空间？
   - 是否需要流水线化或并行化？
   - 有哪些信号可以提前产生（提前一个cycle）？

3. 功耗评估（如需要）
   - 估计的动态功耗（基于切换率）
   - 可能的功耗优化方向

4. 时序裕度评估
   - 预计在目标工艺下能达到的最高频率
   - 频率裕度有多少？
```

### 步骤 7: 仿真交付物生成

```
请生成最终的仿真验证报告，包括：

1. Testbench 代码（含注释）
2. 测试用例列表和描述
3. 仿真结果汇总
   - 功能验证通过/失败统计
   - 性能指标测量结果
   - 覆盖率报告（如有）
4. 性能分析报告
5. 已发现的问题列表和修复说明
6. 推荐的优化方向
```

## ⚠️ 常见问题和处理

### 问题1: 仿真无法编译
**症状**：Verilog 语法错误或信号不匹配
**处理**：
1. 检查 RTL 代码是否有语法问题
2. 检查 Testbench 中的信号连接是否正确
3. 让AI修复代码

### 问题2: 仿真运行出错，出现未知状态(x/z)
**症状**：关键信号为 x 或 z
**处理**：
1. 检查是否所有复位都正确释放
2. 检查是否有浮空信号
3. 检查初始值设置

### 问题3: 性能未达到预期
**症状**：吞吐量或延迟不满足设计目标
**处理**：
1. 分析性能瓶颈在哪里（数据路径？控制逻辑？）
2. 让AI提出优化方案（流水线、并行处理等）
3. 评估优化对面积的影响

### 问题4: 覆盖率不足
**症状**：某些分支未被覆盖
**处理**：
1. 分析未覆盖的分支是什么（错误处理？特殊模式？）
2. 设计针对性的测试用例
3. 让AI补充相应的测试激励

## 💡 最佳实践

1. **尽早仿真**：不要等代码完全完成，部分模块就可以开始仿真
2. **分层验证**：先验证单个模块，再验证子系统，最后验证整体
3. **自动化检查**：使用断言和检查点，而不是手工观察波形
4. **长时间运行**：不要只跑几百个 cycle，要跑 10K+ cycles
5. **保存波形**：仿真通过后保存波形，方便后续参考

---

## 🔄 与下一步的衔接

仿真验证完成后：
1. **代码优化**：基于性能分析，让AI优化代码
2. **综合验证**：进行后仿真（SDF back-annotation）验证
3. **性能评估**：汇总所有指标，输出最终的设计报告
