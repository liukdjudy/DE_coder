# RTL Design AI Skill: 设计交付与文档生成

## 目的
AI生成**完整的设计文档、集成指南和交付物清单**，确保设计可被后续工程师无缝接收。

---

## 前置条件
- [ ] RTL代码已完成并通过代码审查
- [ ] 仿真验证已完成，所有测试通过
- [ ] 性能指标已确认
- [ ] 设计冻结

---

## 📋 核心工作流

### 步骤 1: 设计文档生成请求

**工程师向AI提供信息**：

```
基于已完成的RTL设计和验证，请生成完整的交付文档。

【设计信息汇总】
- 模块名称：[xxx_module]
- 功能简述：[简要描述]
- 最终RTL文件列表：[列表或路径]
- 仿真验证报告位置：[路径]

【需要生成的文档】
请生成以下所有文档：

1. 设计规格说明书 (Design Specification)
   - 功能概述
   - 接口定义（所有输入/输出信号及其含义）
   - 工作模式（如果有多个）
   - 时序规范（时钟频率、复位要求等）
   - 性能指标汇总

2. 微架构设计文档 (Micro-architecture)
   - 整体架构框图（强烈推荐用 ASCII 图形表示）
   - 模块划分说明
   - 数据通路图
   - 控制路径说明
   - 关键信号的时序关系

3. RTL实现指南 (RTL Implementation)
   - 模块文件清单（文件名、行数、功能）
   - 关键模块的内部实现说明
   - 参数配置说明
   - 已知的时序关键路径
   - 代码中的特殊设计考虑

4. 集成使用手册 (Integration Guide)
   - 例化方法（带代码示例）
   - 参数含义详解
   - 信号连接说明
   - 时钟/复位处理要求
   - 常见集成错误及避免方法

5. 验证报告 (Verification Report)
   - 测试概览（覆盖的功能点）
   - 测试结果汇总
   - 性能指标测量结果
   - 已发现和修复的问题列表
   - 仿真覆盖率报告

6. 性能分析报告 (Performance Analysis)
   - 吞吐量分析
   - 延迟分析
   - 功耗估计（如适用）
   - 时序余度分析
   - 优化建议

7. 设计交付清单 (Delivery Checklist)
   - 所有RTL文件
   - 所有文档文件
   - Testbench和仿真脚本
   - IP配置文件（如有）
   - 版本历史和变更记录

【文档格式】
- 推荐格式：Markdown（便于版本管理和审查）
- 可包含的图表：ASCII 流程图、时序图等
- 如需复杂图表：提供 Mermaid 代码或相应的描述

【特殊要求】
- 文档应面向两类读者：集成工程师和流片后验证工程师
- 要明确说明任何已知的限制或注意事项
- 如有多个版本或演进，记录变更历史
```

### 步骤 2: AI生成完整文档

**AI输出内容**（预期）：
```
✓ 设计规格说明书（完整、清晰、有图有表）
✓ 微架构文档（包含框图和数据流说明）
✓ RTL实现指南（每个模块都有详细说明）
✓ 集成使用手册（带真实的代码示例）
✓ 验证报告（汇总所有测试结果）
✓ 性能分析（性能评估和优化建议）
✓ 设计交付清单（完整的文件清单）
```

### 步骤 3: 文档审查和完善

**检查清单**：

#### 3.1 内容完整性
- [ ] 文档是否覆盖了设计的所有关键方面？
- [ ] 接口定义是否清晰完整？
- [ ] 所有参数都有详细说明吗？
- [ ] 是否遗漏了某些重要的设计决策说明？

#### 3.2 准确性
- [ ] 文档中的信息是否与实际RTL代码一致？
- [ ] 信号名、位宽、极性等信息是否正确？
- [ ] 性能数据是否与仿真结果一致？
- [ ] 图表和说明是否准确？

#### 3.3 可读性
- [ ] 文档结构是否清晰？目录完整吗？
- [ ] 用词是否恰当？是否有歧义？
- [ ] 图表是否清晰易理解？
- [ ] 是否有过多的技术细节导致晦涩？

#### 3.4 可用性
- [ ] 使用者（集成工程师）能否快速找到所需信息？
- [ ] 集成说明是否足够详细？
- [ ] 常见问题是否有解答？
- [ ] 有无支持性的参考资料？

**修改反馈给AI**：
```
感谢生成的文档。我进行了审查，有以下修改建议：

【内容补充】
1. [某个部分]缺乏详细说明
   建议：补充 xxxx 内容

2. 在 [某个章节] 中建议添加一个使用示例

【准确性修正】
1. 第 x 页，信号 xxx 的说明与代码不符
   实际情况：xxxx

2. 性能指标中，吞吐量应该是 xxx（当前文档写的是yyy）

【表述改进】
1. [某些技术术语]可以用更简单的方式解释
2. [某个流程图]建议重新组织逻辑

【其他意见】
- xxxx

请基于上述反馈修改文档。
```

- [ ] 发布日期是否更新？
- [ ] 作者信息是否完整？

```

2. 确保所有交叉引用一致
3. 生成一份文档索引（列出所有文档及其用途）

```

## ⚠️ 常见问题和处理

### 问题1: 文档中的信息与代码不一致
**症状**：集成工程师发现文档描述与实际行为不符
**处理**：
1. 检查是否是最新版本的文档
2. 修正文档或代码，保证一致
3. 版本号递增，明确标注变更

### 问题2: 文档过于复杂或过于简化
**症状**：使用者无法快速理解或遗漏重要信息
**处理**：
1. 调整文档深度和组织
2. 添加更多示例或图表
3. 分层编写（概览+详细）

### 问题3: 交付物不完整
**症状**：缺少某些必要的文件或说明
**处理**：
1. 使用交付清单逐一检查
2. 补齐遗漏的文件或文档
3. 再次验证

## 💡 最佳实践

1. **文档与代码同步**：代码改变时，文档也要同步更新
2. **版本管理**：清晰记录每个版本的变更
3. **面向使用者**：写文档时要考虑后续使用者的需求
4. **图表清晰**：使用流程图、时序图等视觉辅助
5. **包含示例**：真实的代码示例比抽象描述更有帮助
6. **定期审查**：设计完成后，再次审查文档准确性
