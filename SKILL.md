## 🎯 核心能力与角色分工

### 工程师职责
- **输入**: 需求、约束、反馈、代码审查
- **决策**: 架构选择、方案评审、质量把关
- **输出**: 评审意见、修改指导、最终验收

### AI助手职责
- **生成**: 方案建议、代码框架、文档内容
- **建议**: 优化方向、最佳实践、问题检测
- **迭代**: 根据反馈快速调整和优化

### 推荐的沟通模式
```
工程师 ──[输入信息+明确要求]──▶ AI
                              │
                        ┌─────▼─────┐
                        │  AI生成    │
                        └─────┬─────┘
                              │
工程师 ◀─[评审意见+修改建议]── AI输出物
   │
   ├─ 满意?
   │  ├─ 是 ──▶ 进入下一阶段
   │  └─ 否 ──▶ 提供反馈,继续迭代
```

## 📖 完整工作流详解

### **阶段1: 需求分析与架构设计** 🏗️

**目标**: 输出经评审通过的架构设计方案

**前置条件检查**:
- [ ] 设计需求规范已使用模板填写完整
- [ ] 所有相关IP文档已收集
- [ ] 编码规范/Review Checklist 已准备

**核心步骤**:

#### 步骤 1.1: 需求信息输入
工程师将以下信息完整提供给AI：

```
【系统信息】
• 项目名称和目标频率
• 主要功能和关键指标
• 面积/功耗约束

【设计需求规范】
[直接复制设计需求模板]

【已有资源】
• IP列表及其指标
• 参考设计或类似项目
• 已确定的编码规范

【关键问题】
需要AI在架构设计中着重考虑的问题
```

**AI预期输出**:
```
✓ 需求完整性分析 (有疏漏?风险?)
✓ 推荐架构方案 + 设计思路说明
✓ 模块划分与接口定义
✓ 关键路径时序分析
✓ IP选择评估报告
✓ 风险识别与缓解方案
✓ 实现策略建议
```

#### 步骤 1.2: 工程师架构评审

使用以下检查清单评审AI生成的架构:

**功能合理性**:
- [ ] 架构是否满足所有功能需求？
- [ ] 接口定义是否清晰完整（信号含义、宽度、时序）？
- [ ] 数据流是否闭环无遗漏？
- [ ] 特殊场景/边界条件的处理是否考虑完整？

**性能可行性**:
- [ ] 时序可行性? 是否需要流水线/并行化?
- [ ] 关键路径上是否过多组合逻辑层级?
- [ ] 频率目标能否达到?

**IP和资源**:
- [ ] IP选择是否合理? 有更好的替代方案吗?
- [ ] 面积和功耗预估是否合理?
- [ ] IP集成复杂度是否可控?

**设计鲁棒性**:
- [ ] 复位、时钟管理方案是否完整?
- [ ] 跨时钟域设计是否考虑（如有）?
- [ ] 可测性（DFT）是否有初步考虑?

## 🔄 跨阶段迭代管理

### 问题反溯与修复流程

当某个阶段发现问题时，可能需要回到前面的阶段：

```
发现问题
    ↓
【问题类型】
    ├─ 架构缺陷 ──→ 回到阶段1，修改架构设计
    ├─ 代码缺陷 ──→ 回到阶段2，修改RTL代码
    ├─ 验证不完整 ──→ 回到阶段3，补充测试场景
    └─ 文档错误 ──→ 回到阶段4，修正文档

修复后 ──→ 重新执行涉及的后续阶段验证
```

### 质量检查关键节点

| 节点 | 何时执行 | 责任人 | 检查内容 |
|-----|--------|-------|--------|
| 架构冻结评审 | 阶段1末 | 工程师 | 需求满足 + 可行性 |
| 代码审查 | 阶段2末 | 工程师 | 功能 + 规范 + 时序 |
| 验证通过评审 | 阶段3末 | 工程师 | 所有测试通过 + 性能达标 |
| 交付完整性检查 | 阶段4末 | 工程师 | 交付清单 ✓ |

## 📚 最佳实践与常见问题

### ✅ 最佳实践

1. **清晰的输入信息**
   - 不要省略细节，完整的需求信息能大幅提高AI输出质量
   - 提供参考设计或已有的类似实现

2. **定期迭代反馈**
   - 不要期望一次到位，3-5次迭代是正常的
   - 每次反馈要具体指出问题位置和期望

3. **质量把关**
   - 工程师的评审决策权最高，不要盲目接受AI建议
   - 标记"可选"的优化项，优先修复功能和规范问题

4. **文档同步**
   - 代码修改后立即更新对应的文档
   - 保持架构文档、RTL代码、Testbench三者一致

### ⚠️ 常见问题处理

**问题1: 架构方案不满足时序需求**
```
症状: AI给出的架构在目标频率下无法实现
处理:
1. 检查关键路径上是否有过多组合逻辑
2. 考虑增加流水线级数
3. 考虑并行化处理多个请求
4. 与AI讨论具体的时序瓶颈
```

**问题2: 代码综合后报警告/错误**
```
症状: RTL代码有综合工具警告或综合失败
处理:
1. 检查是否有unsynthesizable的语法
2. 检查是否有不定时序的问题
3. 反馈给AI具体的警告信息，要求修改
```

**问题3: 验证覆盖不完整**
```
症状: 代码跑仿真后发现新的bug
处理:
1. 分析bug对应的未覆盖场景
2. 补充该场景的测试用例
3. 反馈给AI完善Testbench
```
