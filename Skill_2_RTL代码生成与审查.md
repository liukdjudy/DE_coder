# RTL Design AI Skill: RTL代码生成与代码审查

## 目的
基于评审通过的架构方案，与AI协作生成**高质量的RTL代码**，并进行代码审查。

---

## 前置条件
- [ ] 架构方案已获得评审通过
- [ ] 编码规范文档已准备（RTL_Coding_Style.md）
- [ ] 代码Review Checklist已准备
- [ ] IP模块已集成或可用

---

## 📋 核心工作流

### 步骤 1: RTL代码生成请求

**工程师向AI提供信息**：

```
你是一名资深RTL设计工程师，现在基于已评审通过的架构方案进行RTL代码生成。

【架构方案信息】
已评审通过的架构文档见：[架构文档路径或内容]

【编码规范】
请严格遵循以下编码规范：

[直接粘贴 RTL_Coding_Style 的关键要点]

关键规范摘要：
- 模块名命名：snake_case_vX.Y
- 信号名：snake_case
- 参数名：UPPER_CASE
- 复位：统一使用 active-low 异步复位
- 所有 flip-flop 都必须有复位信号
- 禁止跨时钟域直接连接（使用CDC设计）
- 优先使用同步设计

【设计要求】
- 目标频率：[具体MHz]
- 最大面积预算：[具体值或相对值]
- 功耗约束：[动态/漏电]
- 编码语言：Verilog / SystemVerilog

【请执行以下任务】
1. 基于架构方案，生成完整的顶层模块代码
2. 生成关键的子模块代码（优先级：数据路径 > 控制路径 > 外围逻辑）
3. 对每个模块和关键逻辑给出设计解释
4. 提出可能的综合、时序问题及优化建议
5. 列出所有外部IP的例化方式和参数

【代码生成策略】
- 逐个模块生成，每个模块完整可仿真
- 每个代码块都要有详细的注释
- 代码必须能通过语法检查和基本功能仿真
```

### 步骤 2: AI生成RTL代码

**AI输出内容**（预期）：
```
✓ 顶层模块完整代码（带参数化、接口定义）
✓ 关键子模块代码（如控制逻辑、仲裁器等）
✓ 每个模块的设计说明文档
✓ 综合和时序分析预测
✓ 关键信号的时序时序图（如需要）
✓ 推荐的验证激励思路
```

### 步骤 3: 工程师代码审查 - 第一轮

**使用以下 Code Review Checklist**：

#### 3.1 功能正确性
- [ ] 模块接口是否与架构文档一致？
- [ ] 状态机/时序逻辑是否正确？
- [ ] 数据路径是否完整无漏洞？
- [ ] 边界条件处理是否完整？
- [ ] 异常情况（如：满/空、溢出）处理是否覆盖？

#### 3.2 编码规范遵循
- [ ] 命名规范是否统一？
- [ ] 是否有 latch？
- [ ] 所有 flip-flop 是否都有异步复位？
- [ ] 复位初值是否合理？
- [ ] 是否有跨时钟域未同步的信号？
- [ ] 注释是否充分？

#### 3.3 时序和综合
- [ ] 组合逻辑的深度是否过深？（关键路径预判）
- [ ] 是否有大的 mux 树或优先级编码器？
- [ ] 寄存器更新是否在时钟上升沿？
- [ ] 异步复位是否正确使用？
- [ ] 有没有不必要的多路选择器层级？

#### 3.4 代码可读性和可维护性
- [ ] 模块大小是否过大？（>1000行建议拆分）
- [ ] 参数化是否充分？
- [ ] 魔法数字是否都用参数/宏定义？
- [ ] 代码结构是否清晰、易于理解？

#### 3.5 工程化考虑
- [ ] 是否预留了 debug 信号/信号？
- [ ] 是否考虑了版本控制和兼容性？
- [ ] 是否有 TODO 或未完成项？
- [ ] 集成时是否考虑了各种配置？

**反馈给AI**：
```
感谢代码生成。我进行了初步审查，有以下问题需要修正：

【Critical Issues】（必须修改）
1. [问题1]：位置在 xxx 文件 line xxx
   问题描述：xxxx
   建议修改：xxxx

2. [问题2]：...

【Major Issues】（强烈建议修改）
1. [问题]：...

【Minor Issues / 建议】
1. [问题]：...
   可选修改

【其他意见】
- xxxx

请基于上述反馈修改代码。
```

### 步骤 4: AI代码优化与修正

AI修改代码后，重复步骤 3-4，直到审查通过。

**加速审查的技巧**：
- 第一轮审查时可以集中指出几个代表性的问题，让AI理解风格
- 后续改进会更加贴切
- 对明显的风格问题，直接给出修改示例

### 步骤 5: 最终代码检查与集成准备

**最终检查清单**：
- [ ] 所有文件都符合命名规范？
- [ ] 没有未使用的信号或参数？
- [ ] 是否有 TODO 注释未完成？
- [ ] 代码的行数是否合理？
- [ ] IP例化部分是否正确？
- [ ] 顶层接口文档是否完整？

**生成交付物**：
```
请生成最终交付的代码包，包括：

1. 所有 .v/.sv 文件（含注释和设计说明）
2. 顶层模块的接口定义文档
3. 参数配置说明文件
4. 集成指南（如何例化、参数含义等）
5. 已知限制和注意事项
```

## ⚠️ 常见问题和处理

### 问题1: 生成的代码时序不满足
**症状**：AI生成的代码无法在目标频率下综合
**处理**：
1. 具体指出哪条关键路径超标
2. 让AI添加流水线或使用快速逻辑
3. 讨论是否需要重新调整架构

### 问题2: 代码中有大量逻辑错误
**症状**：仿真发现多处逻辑bug
**处理**：
1. 挑选3-5个典型错误，给出详细反馈
2. 让AI理解问题的本质（而非逐个修复）
3. 重新生成相关代码段

### 问题3: 生成的代码过于复杂
**症状**：代码冗长、嵌套深、难以理解
**处理**：
1. 让AI简化代码，使用更直观的方式
2. 提出具体的简化建议
3. 要求分模块重写

### 问题4: 代码量过大，难以一次性审查
**症状**：生成了数千行代码
**处理**：
1. 让AI分模块生成，每次生成一个模块
2. 逐个审查和改进
3. 最后集成

## 💡 最佳实践

1. **分阶段生成**：不要一次性要求全部代码，按模块生成和审查
2. **清晰的反馈**：指出问题的精确位置和理由，不要笼统地说"重写"
3. **代码示例**：对某些问题，可以直接给出修改示例，帮助AI理解
4. **权衡考虑**：性能 vs 面积 vs 复杂度，明确优先级
5. **及时测试**：生成的代码最好立即进行简单的仿真验证
